同意管理アーキテクチャ — 法制度の基礎から実装まで

なぜ Liverty Music の現在の spec は不十分か、業界はどう解決しているか。法令テキスト → 業界フレームワーク → 各ベンダー実装の順で網羅

1 行サマリ

結論 個人情報の越境提供を伴う分析ツール利用には、サーバーサイドに同意状態を永続化し、データ送信前に同意確認するのが世界の標準パターン。ブラウザ localStorage だけでは GDPR Article 7 / APPI 28条が要求する「実効的な同意」を満たせず、また「いつでも撤回可能」の要件も BE 経路で破綻する。

1. 法制度の基礎

1.1 GDPR Article 7 — 同意の 4 つの柱

EU 一般データ保護規則 Article 7 は同意の有効性を 4 条件で定義しています。これは世界の個人情報保護法 (CCPA, LGPD, APPI 等) の事実上のモデルになっています。

原則意味UI/UX への含意
Freely given
(自由意志)		サービス利用と引き換えに同意を強要されてはならない同意拒否でも基本機能は使える設計
Specific
(具体的)		目的ごとに別個の同意が必要「分析」「マーケ」を別トグルにする (1つの「全部 OK」は不可)
Informed
(情報提供あり)		誰に何の目的でデータが渡るか開示する受信者 (PostHog 等)、国名、保存期間、撤回方法を明示
Unambiguous
(明確な意思表示)		"clear affirmative act" — 積極的な行為のみ有効pre-checked box (opt-out) は NG、未操作の continue も NG
Article 7(3) の重要規定 「同意を撤回するのは与えるのと同程度に容易でなければならない」。つまり workflow の 4 ステップ先で見つかる settings > advanced ではダメ。

さらに 撤回は遅滞なく反映される必要がある。フロントエンドだけ stop して BE は送り続ける構成は Article 7(3) 違反の典型例。

1.2 GDPR Article 25 — Privacy by Design and by Default

条文の要点

製品設計の段階から、デフォルトでプライバシー保護的になるよう実装する義務。「同意したらデフォルトで全部 ON」ではなく、「同意なしならデフォルトで全部 OFF」が要求される。

これが「opt-in (default OFF) が業界デフォルト」「opt-out (default ON) は legal risk」と言われる根拠条文。

1.3 APPI (日本) 第28条 — 越境移転の特則

日本の個人情報保護法第28条は、「外国にある第三者」への個人データ提供に関する特則です。Liverty Music は PostHog Cloud EU (オランダ) を使うため、毎回の data transfer がこの条文の対象になります。

論点APPI 28条の要件
原則越境提供前に 本人の事前同意が必要 (= opt-in)
同意取得時の提供情報(a) 当該国名
(b) 当該国の個人情報保護制度
(c) 受領者の保護措置
例外 1: 適切性認定国2019/01 から EU と英国が "Japan と同等" として認定 → 本人同意不要
例外 2: 同等水準の保護措置を講じる事業者毎年の certification 取得が必要
2026年改正統計目的・非識別化処理は同意不要に → ただし「非識別化の技術的・組織的措置」を文書化する義務
EU が例外 (a) で認定されているのに、なぜ Liverty Music で 28条が問題になるのか?

PostHog は EU 内のデータセンターを使っているが、運営会社 Klant Solutions B.V. はオランダ法人で、データの実際の処理者 (controller/processor) です。APPI 28条が対象とするのは「外国にある第三者」であり、EU の場合は「適切性認定により事前同意は不要、ただし提供主体 (Liverty) は受領者の保護措置の継続的確保義務がある」となります。

つまり 本人同意は法的には必須ではないけれど、Liverty Music が "Article 28 explicit consent" を design.md に書いている以上、それは 自社の commitment として実装上守る必要がある (commitment を破る spec を出版することの reputation/audit リスク)。

1.4 三法の対応関係まとめ

GDPR Article 7 APPI 28条 Liverty Music の現状 ──────────── ──────── ──────────────────── Freely given 自由意志 ✅ "Set up later" あり Specific 目的別 ✅ 2 トグル分離 (Round 1+2) Informed 国名/受領者明示 ⚠️ PostHog 名前あるが 監査 trail なし Unambiguous 積極的意思表示 ✅ opt-in (Round 1+2) Article 7(3) 撤回 継続的保護義務 ❌ FE 撤回が BE 経路に 伝播しない (Round 4 の発見) Article 25 Privacy by Design ⚠️ BE が同意を構造的に Privacy by Default 知らない設計

2.2 Google Consent Mode v2

Google が 2024年3月から導入した同意管理規格。EEA (EU + 英国) ユーザー向けに Google サービスを使う場合は事実上必須。

Consent Type意味
ad_storage広告関連 Cookie の保存
analytics_storage分析 Cookie の保存
ad_user_dataユーザーデータの Google 広告への送信
ad_personalizationパーソナライズ広告

2.3 サーバーサイド Consent Mode の核心

2024年 Google のサーバーサイド GTM ガイダンスより:

Google 公式の引用
"Consent is no longer automatic — the server has no idea what the user chose unless you explicitly send it. When an event fires, the request must include the current consent state — in the payload, headers, or a first-party cookie. The server reads consent and maps the incoming consent into a variable."

これは Liverty Music の Round 4 で bot が指摘した問題そのもの。NATS event → analytics-consumer → PostHog の経路で、event payload に consent 情報が乗っていない (or BE 側に DB がない) ため、server が同意を知る術がない。

3. 各ベンダーの公式ガイドライン

3.1 PostHog (Liverty Music 採用ベンダー)

3.1.1 公式ドキュメントの抜粋

PostHog 公式 (privacy/data-collection)
"If a user opts out then you must stop all data capturing and processing. It is up to you to ensure that your application logic either does not load PostHog SDKs or disables data capturing when a user opts out."
PostHog 公式 (privacy/gdpr-compliance)
"Always gate user-data captures on a consent check. If you forget to gate, you're processing personal data without a legal basis, which is the kind of thing GDPR and CCPA enforcement actions tend to focus on."

3.1.2 PostHog が提供する API

API用途Liverty Music で使う場面
opt_out_capturing_by_default: trueSDK 初期化時にデフォルト OFF必須 Decision 7 の opt-in 方針と一致
cookieless_mode: 'on_reject'同意拒否時は Cookie を一切セットしない必須 pre-consent allowlist の実装
posthog.opt_in_capturing()同意後に SDK を有効化consent screen の OK ボタン
posthog.opt_out_capturing()撤回時に SDK を無効化settings の OFF
posthog.has_opted_in_capturing()同意状態を checkすべての capture() の前で確認
posthog.identify()ユーザー識別同意前は呼ぶな identify は PII 生成行為と見なされる

3.1.3 PostHog が "serverside で必要" と明示する事項

PostHog 公式 (privacy/data-collection)
"For server-side implementations, you need to manage the opt-out process manually. The server is responsible for generating IDs, maintaining ID persistence, and managing the opt-out state of users."

つまり PostHog 自身が「サーバーサイドでは opt-out 状態を管理する責任は実装側にある」と明記している。Liverty Music の analytics-consumer はこの責任を果たす必要がある。

3.2 RudderStack (CDP デファクト) のパターン

RudderStack はイベント駆動分析の業界標準的な実装を提供しており、その consent 管理方式は多くの組織が参考にしています。

3.2.1 consentManagement object をイベントペイロードに埋め込む

// RudderStack のイベントフォーマット
{
  "event": "ticket.purchase.completed",
  "userId": "uuid-...",
  "properties": { ... },
  "context": {
    "consentManagement": {
      "allowedConsentIds": ["analytics", "marketing"],
      "deniedConsentIds": []
    }
  }
}

3.2.2 Destination 側でフィルタリング

RudderStack 公式
"You may want to send data to a marketing tool only if a user has opted in to advertising and marketing tracking. In this case, you would configure your marketing destination to require allowedConsentIds for both analytics and marketing. If event payloads do not include these consent IDs, RudderStack will drop them, ensuring you don't send non-compliant user data."

Liverty Music への含意: NATS event 自体に同意 ID を埋め込む方式 vs analytics-consumer が DB を見に行く方式の 二択がある。前者は event payload が大きくなる、後者は DB ラウンドトリップが増える。

3.3 Segment (Twilio)

Segment はサーバーサイド分析の老舗。Liverty Music の analytics-consumer に最も近いアーキテクチャ。

同等パターン

  • Privacy / Consent Settings: ワークスペース単位で「どの categories の consent が必要か」を destination ごとに設定
  • Event-level consent: track() / identify() の calls に consent context を渡せる
  • サーバーサイド SDK: Node.js / Python / Go の各 SDK で同等の API 提供

Segment が公式に推奨するパターン: "Consent should be persisted to your application database, not relied on browser storage alone, for server-side analytics paths."

3.4 Mixpanel

API用途
mixpanel.opt_out_tracking()SDK レベルで送信停止
mixpanel.has_opted_out_tracking()同意状態 check
opt_out_tracking_by_default: trueデフォルト opt-out
サーバー側 API各 SDK で同等の opt-out 管理 API

Mixpanel は ユーザー単位の opt-out リストを Mixpanel 側で管理する仕組みも提供 (Service Accounts API)。これにより BE が opt-out ユーザーを Mixpanel に登録すると、それ以降の事故的送信もブロックされる二重化が可能。

6. spec に追加すべき要件 (推奨)

1
analytics-consent capability に新規 Requirement 追加
"Consent state SHALL be persisted to the backend database via a dedicated RPC."
理由: GDPR Article 7 (Informed/Unambiguous) + APPI 28条 commitment
2
analytics-consent capability に撤回の BE 即時反映 Requirement 追加
"Revoking consent SHALL update the backend consent record within the same RPC call. analytics-consumer SHALL observe the revocation on its next event."
理由: GDPR Article 7(3) "撤回の容易性"
3
product-analytics capability に consumer 側の同意確認 Requirement 追加
"The analytics-consumer SHALL consult the backend consent record before forwarding any event with a non-empty distinct_id. Events for users without analytics consent SHALL be dropped (and optionally logged for audit)."
理由: PostHog 公式の "Always gate user-data captures on a consent check"
4
pre-consent BE event (user.created 等) の deferred 処理
"BE events emitted before the user reaches the consent screen (user.created, account.signup.completed) SHALL be queued. On consent grant, the queued events SHALL be flushed to PostHog. On decline, dropped."
理由: Google Consent Mode v2 (2024/10 から導入された deferred replay) を参考
5
Decision 7 / Open Question 3 の更新
「BE-local consent persistence は 2026年業界標準 (Transcend, PostHog 公式, RudderStack デファクト) および CNIL 多デバイスガイダンスに準拠した選択である」と明記
6
tasks.md に Batch 2b/3 タスク追加
DB migration (user_consents テーブル), ConsentRepository, analytics-consumer 側の consent check, 撤回の RPC, deferred event の queue

8. 推奨

業界標準に準拠した spec 修正を強く推奨

理由:
  • PostHog 公式が "Always gate" を明文化しており、これを spec で守らないと vendor の使用契約上も問題
  • APPI 28条を framing として使うなら "industry standard alignment" は法的にも最強
  • 2026年は server-side consent architecture が industry default に。今ここを正しくしないと Batch 3 で同じ議論を再度することになる
  • Round 5 で bot がさらに何か見つける可能性は残るが、ここまでの finding 性質を見ると規模は小さくなる傾向 (cascading propagation の最終段階に近い)
