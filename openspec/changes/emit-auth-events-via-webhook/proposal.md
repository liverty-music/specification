## Why

The archived `introduce-analytics-tool` change shipped the full conversion funnel but deliberately descoped its two account-auth signals (Decision 15): `account.signup.completed` was redundant with the already-emitted `user.created`, and `account.login` had no clean backend hook — the only login-adjacent touchpoint, the `pre_access_token` webhook, fires on *every* access-token mint including silent refresh-token grants, so emitting there over-counts logins.

The product owner now wants login instrumented anyway: `account.login` is the single best **active-user / returning-user signal** for retention cohorts (D7/D30 by signup month already exist, but they need a recurring "this user came back" event to be meaningful). Decision 15's *deferral* is reversed; its *reasoning* is not. This change therefore MUST solve the over-count problem (a refresh is not a login) and MUST avoid re-introducing the signup redundancy — emitting a misleading login count or a double-counted signup is worse than emitting nothing.

**This change was implemented, shipped to prod, and REVERTED (2026-07-02).** The first attempt sourced `account.login` from an Actions v2 **`response` execution** on `/zitadel.session.v2.SessionService/CreateSession`. That execution type does not *observe* the API call — it **replaces the API response with the webhook's return body** (it is not observe-only; `interruptOnError:false` does not change this). The backend returned `{}`, which stripped `sessionId` from the `CreateSession` response; the Login UI v2 then called `GetSession("")`, and **every new interactive sign-in failed instance-wide** until the Target/Execution was destroyed (`pulumi cancel` + `destroy --target`). Two build-time assumptions were falsified in prod: (1) that a `response` execution could observe without mutating, and (2) that the login user was reachable at `request.checks.user.userId` on `CreateSession` — in the hosted Login UI v2 flow the user is attached by a later `SetSession`, so every call logged `payload missing request.checks.user.userId`. This proposal supersedes that approach.

## What Changes

- Emit `account.login` server-side **once per user-initiated login**, attributed to the platform `UserId` (UUID), explicitly NOT on token refresh.
- Re-source the login signal from a fire-and-forget Actions v2 **`event` execution** (condition `event.event = "<login-event-type>"`). An `event` execution fires **after** the login event is persisted to the Zitadel eventstore, receives the stored event payload (`{aggregateID, event_type, userID, event_payload}`), and **cannot manipulate any API request or response** — so it structurally cannot break sign-in the way the reverted `response` execution did. This is Zitadel's documented pattern for reacting to events / side-effects.
  - Docs backing this pattern: [Actions v2 concepts](https://zitadel.com/docs/concepts/features/actions_v2), [Actions v2 usage](https://zitadel.com/docs/guides/integrate/actions/usage), [Test Actions Event](https://zitadel.com/docs/guides/integrate/actions/testing-event). Our Zitadel is v4.7.1+, so Actions v2 Executions are GA.
- **Determine the exact `event_type` empirically — do NOT guess.** Discover it via the Zitadel **Events API** (`ListEvents`, `POST /admin/v1/events/_search`): exercise a fresh interactive login vs a silent `refresh_token` grant and pick the event type that fires **once per user-initiated login**, does **not** fire on refresh, and carries the Zitadel `userID` in its aggregate/payload. The recorded query result is the justification for the type chosen.
- **Discard the `response`/`request` execution approach entirely** for this signal. Those execution types replace the API payload with the webhook return (the cause of the 2026-07-02 outage) and offer no observe-only mode. Only an `event` execution is used.
- Verify the webhook body the same way the deployed `pre_access_token` handler does — the Target stays `PAYLOAD_TYPE_JWT` and the backend validates the JWT signature against the Zitadel instance JWKS (`WebhookValidator`). No shared HMAC secret is introduced; JWKS signature + the private-only webhook listener is the security boundary, identical to the existing Target.
- Read the login user identifier as the Zitadel `userID` carried directly in the event payload (the event's aggregate is the user), then map that to the platform `UserId` via the existing `GetByExternalID` lookup, because `AnalyticsClient.Enqueue` requires the platform `UserId` as `distinct_id` (never the IdP subject).
- Wire the analytics path **non-fatal and non-blocking**: the handler publishes a domain event to NATS and lets the `analytics-consumer` forward it to PostHog, matching the existing publisher → consumer → PostHog pattern. The `event` execution is fire-and-forget, so the handler's response is irrelevant to Zitadel and can never affect login.
- **Reuse the backend plumbing already deployed (orphaned) in v1.18.0**: the `ACCOUNT` JetStream stream, `SubjectAccountLogin` (`ACCOUNT.login`), `AnalyticsConsumer.HandleAccountLogin`, its consumer subscription, and the KEDA trigger remain valid. Only the webhook handler's **payload parsing** changes — from the `CreateSession` JWT (`request.checks.user.userId`) to the `event`-execution payload (`userID`).
- Resolve `account.signup.completed` by **aliasing signup analytics to the existing `user.created`** — no new event is emitted. The unused `EventAccountSignupCompleted` constant remains a no-op alias documented as such; `user.created` is the canonical signup signal.

## Capabilities

### Modified Capabilities

- `product-analytics`: account authentication events are now emitted server-side. `account.login` is login-specific (one event per user-initiated login, never on token refresh) and attributed to the platform `UserId`; signup continues to be represented by `user.created` with no duplicate `account.signup.completed`.

## Impact

- **Empirical `event_type` discovery first**: a read-only `ListEvents` query on Zitadel (fresh login vs refresh) decides the exact event type; the recorded result is the justification for the type chosen. No prod-mutating step precedes it.
- **Zitadel Actions v2 reconfiguration (cloud-provisioning)**: replace the destroyed `response`-condition Execution on `CreateSession` with an `event`-condition Execution bound to the discovered login `event_type`, pointing at the (retained) `PAYLOAD_TYPE_JWT` Target with `interruptOnError: false`. This needs a new `ZitadelExecutionEvent` dynamic resource (`condition: { event: { event } }`); the `ZitadelExecutionResponse` resource added by the reverted attempt is removed. No `request`/`response` execution is created for this signal.
- **Backend handler payload change (reuse otherwise)**: the deployed webhook handler is adapted to parse the `event`-execution payload and read `userID` directly (not the JWT-nested `request.checks.user.userId`). Its JWKS verification, route registration on the internal-only listener, `EventPublisher` wiring, and the `userID → UserId` lookup are reused.
- **Reused NATS subject + stream + consumer method (already deployed in v1.18.0)**: `ACCOUNT.login` on the `ACCOUNT` stream, `AnalyticsConsumer.HandleAccountLogin`, its subscription, and the KEDA trigger (stream `ACCOUNT`, consumer `ACCOUNT_login`) remain as shipped — no new messaging surface.
- **User lookup on the login path**: `userID → UserId` via the existing `GetByExternalID`, run per login event only (the `event` execution never fires on refresh, so no refresh query occurs).
- **No new RPC, no proto change, no DB schema change**: this rides the existing Zitadel Actions webhook + NATS + analytics-consumer + PostHog infrastructure. The specification #684 event-catalog delta is retained and corrected to describe the `event`-execution source.
- **No frontend change**: `account.login` is BE-sourced / trust-critical per the event catalogue; the frontend does not emit it.
- **Event catalogue**: `account.login` moves from "defined but no publisher" to "emitted (BE, source = Zitadel `event` execution, refresh-excluded)"; `account.signup.completed` is documented as an alias of `user.created` (not separately emitted).
- **Risk surface**: the outage-causing risk is *eliminated by construction* — an `event` execution cannot alter the `CreateSession`/`SetSession` response, so it cannot break sign-in. The remaining risks are (a) picking the wrong `event_type` (mitigated by empirical discovery + a refresh-exclusion test) and (b) best-effort delivery bounding login-count completeness (acceptable under-count). Both are addressed in design.md.
