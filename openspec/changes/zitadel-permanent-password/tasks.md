- [ ] 1.1 Add `cloud-provisioning/src/zitadel/dynamic/permanent-password.ts`. Model it on `smtp-activation.ts`. Export `ZitadelHumanUserPasswordPermanentArgs` (with `domain: pulumi.Input<string>`, `jwtProfileJson: pulumi.Input<string>`, `userId: pulumi.Input<string>`, `password: pulumi.Input<string>`), the provider object (`permanentPasswordProvider`), and the `ZitadelHumanUserPasswordPermanent extends pulumi.dynamic.Resource` class. The class constructor MUST bake `additionalSecretOutputs: ['password', 'jwtProfileJson']` into the `CustomResourceOptions` it passes to `super(...)`, merging with any caller-supplied `additionalSecretOutputs`. This guarantees state-side encryption even if a future caller forgets to pass the option (`pulumi.secret(...)` inputs do NOT auto-propagate across the dynamic-resource boundary into outputs).
- [ ] 1.2 Implement `create()`: use `zitadelApiCall({ domain, profile, method: 'POST', path: '/management/v1/users/{userId}/password', body: { password, noChangeRequired: true } })`. Treat any 2xx as success. Inspect the failure body for an "already permanent" / "no change" marker (mirroring `isAlreadyActive()` in `smtp-activation.ts`) and treat it as success too — verify the exact response shape during §4 live-ops verification. Return `{ id: 'permanent-password:{userId}', outs: { ...inputs, markedAt: new Date().toISOString() } }`.
- [ ] 1.3 Implement `update()`: re-POST `SetPassword` with `news.password` and `noChangeRequired: true`. Under the call-site directives `ignoreChanges: ['password']` + `replaceOnChanges: ['userId']` (see §2.2), this handler is only reachable when `domain` or `jwtProfileJson` changes — it acts as a drift-recovery / re-assertion path (e.g., if an operator manually clears the permanence flag in Zitadel admin console, the next provider rotation re-marks it). Password rotation itself does NOT pass through `update()`; it cascades from `pulumi up --replace` on the HumanUser into a fresh `create()` via `replaceOnChanges`. Preserve `markedAt` from `olds` for diff stability.
- [ ] 1.4 Implement `delete()` and `read()` as no-ops. `delete()` does NOT call the Management API (no inverse verb exists, and removing the marker should not flip the user back to `changeRequired = true`). `read()` returns the current state unchanged (Zitadel exposes no `GET /password/state` for drift detection).
