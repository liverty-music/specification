## Why

The platform stores two Zitadel `MachineUser` JWT private keys in GCP Secret Manager (GSM):

1. **`zitadel-admin-sa-key`** — bootstrap admin key, generated by Zitadel itself on first boot, written to GSM by the in-pod `bootstrap-uploader` sidecar, consumed by Pulumi to provision Project / Application / Actions v2 resources for the `pulumi-admin` MachineUser.
2. **`zitadel-machine-key`** — Pulumi-provisioned `MachineKey` for the `backend-app` MachineUser, consumed by the backend Go service to call Zitadel Management APIs (sign-up user creation, verification email resend).

The two names follow inconsistent conventions:

- `zitadel-admin-sa-key` uses `<provider>-<principal>-sa-key`, where `-sa-` is a generic credential-type marker but the principal label `admin` does not match its Pulumi resource id `pulumi-admin`.
- `zitadel-machine-key` uses `<provider>-<credential-type>-key` with **no principal binding** — ambiguous now that the platform manages two MachineKeys.

During the §13.15 incident chain (`fix(zitadel): refresh backend-app MachineKey to align with self-hosted`), this ambiguity directly cost time: the operator had to grep call sites to confirm which Zitadel principal the GSM secret was bound to before force-replacing it.

Additionally, the v1 candidate name `zitadel-backend-app-key` does **not** disambiguate which identity system's credential is stored — `backend-app` is intentionally used as the name across three identity systems (K8s ServiceAccount, GCP IAM ServiceAccount, Zitadel MachineUser), all representing the same logical identity. A reader could plausibly interpret `zitadel-backend-app-key` as a GCP IAM key for the backend-app GSA.

This change introduces a single uniform convention:

```
zitadel-machine-key-for-<principal>
```

where `machine-key` mirrors the Zitadel/Pulumi resource type (`zitadel.MachineKey`) and `for-<principal>` makes the binding direction explicit ("the MachineKey FOR principal X"). The pattern:

- Reads naturally as English ("Zitadel MachineKey for backend-app").
- Extends cleanly to any future MachineUser without recasting the convention.
- Restores symmetry between the admin and backend credentials.
- Resolves the `backend-app` overload — the `for-` preposition signals that the suffix is the *owning principal*, not the *consuming system*.

## What Changes

- **The platform SHALL adopt the naming convention `zitadel-machine-key-for-<principal>`** for all GSM secrets that store a Zitadel `MachineKey`.
- **Renames in scope**:
  - `zitadel-machine-key` → `zitadel-machine-key-for-backend-app`
  - `zitadel-admin-sa-key` → `zitadel-machine-key-for-pulumi-admin`
- **Consequential renames in backend** (because the GSM secret name is the file basename mounted into the pod):
  - Env var: `ZITADEL_MACHINE_KEY_PATH` → `ZITADEL_MACHINE_KEY_FOR_BACKEND_APP_PATH`
  - Mounted file path: `/secrets/zitadel/zitadel-machine-key.json` → `/secrets/zitadel/zitadel-machine-key-for-backend-app.json`
  - Go `Config` field (Server + Consumer): renamed to match the new env var via its `envconfig` tag.
- **Migration strategy**:
  - **`zitadel-machine-key-for-backend-app`**: **6-step zero-downtime split** (separate PRs per step, cross-repo coordination required). Steps add the new GSM secret + new env var alongside the old, switch the deployment to read the new file, drop the old fallback, soak 7 days, then destroy the old GSM secret.
  - **`zitadel-machine-key-for-pulumi-admin`**: **3-step migration**. No running pods consume this — only `bootstrap-uploader` (writer) and Pulumi (reader at apply time). `bootstrap-uploader` writes both names during transition; Pulumi switches to read the new name; old is destroyed.
  - The two migrations are independent and MAY run in parallel.
- **Cross-repo coordination** SHALL be enforced via change tasks: each PR declares its predecessor PR in the description; downstream PRs SHALL NOT merge until the predecessor is deployed in dev.
- **Pulumi MachineKey URN** SHALL be renamed in tandem: `zitadel.MachineKey('backend-app-key')` → `zitadel.MachineKey('machine-key-for-backend-app')`, using Pulumi `aliases: [{ name: 'backend-app-key' }]` to map the old URN onto the new one so **no `create-replace-delete` cycle occurs**. A follow-up PR removes the alias after a ≥ 14-day soak.
- **Pulumi MachineUser URNs** (`backend-app`, `pulumi-admin`) SHALL remain unchanged — they already match the Zitadel `userName` 1:1 and reshaping them would break that correspondence.

## Capabilities

### Modified Capabilities

- **`zitadel-self-hosted-deployment`**: Two requirements referencing the legacy GSM secret names SHALL be updated:
  - "Bootstrap Admin Machine Key Stored in Secret Manager" — switch reference from `zitadel-admin-sa-key` to `zitadel-machine-key-for-pulumi-admin`.
  - "Backend MachineKey Lifecycle Tied to Zitadel-Side Identity" — switch reference from `zitadel-machine-key` to `zitadel-machine-key-for-backend-app`.

  Additionally, a new requirement "GSM Naming Convention for Zitadel MachineKey Credentials" SHALL be ADDED to codify the `zitadel-machine-key-for-<principal>` pattern as a normative convention (not just an outcome of this rename).

- **`identity-management`**: The "Retain Break-glass Machine User" requirement SHALL be updated to reference `zitadel-machine-key-for-pulumi-admin` instead of `zitadel-admin-sa-key` (4 occurrences). The break-glass semantics do not change.

Both deltas are authored as `MODIFIED` / `ADDED` Requirements blocks against the landed `specs/`. The `self-hosted-zitadel` change archived on 2026-05-11, so these run as follow-ups against the landed specs.

**Resources**

- During transition, GSM holds both `zitadel-machine-key` and `zitadel-machine-key-for-backend-app`, and both `zitadel-admin-sa-key` and `zitadel-machine-key-for-pulumi-admin`. After full cutover, only the new names remain.
- ESO `ExternalSecret` for backend is updated twice (add new entry, then remove old) for granular rollback.

**Reversibility**

- Each PR is reversible by reverting the prior PR. Old GSM secrets stay alive until the final destroy step (step 6 for backend-app, step 3 for pulumi-admin). Any rollback before the destroy step is a single revert.
- Once a destroy step is merged, rollback requires recreating the old GSM secret — possible but tedious. Mitigation: the destroy step SHALL be merged only after a soak period (7 days for backend-app, one successful Pulumi apply cycle for pulumi-admin).

**Out of scope**

- Renaming the K8s ServiceAccount `backend-app` to disambiguate from the Zitadel MachineUser `backend-app`. The two are intentionally aligned (same logical identity in different ID systems); a separate change MAY revisit this.
- Renaming the Pulumi `MachineUser` URNs (`backend-app`, `pulumi-admin`). They already match the Zitadel `userName` 1:1. **The `MachineKey` URN rename is in scope** — see *What Changes*.
- Tighter coordination with `k8s-naming-cleanup` beyond a shared maintenance window. The two changes touch disjoint files and have different risk profiles (atomic-with-downtime vs zero-downtime split).
