## Context

The dev Cloud SQL instance uses PSC-only connectivity (no public IP). The previous approach using Pulumi's `@pulumi/postgresql` provider to manage PostgreSQL schema and permissions failed because the provider cannot connect from local machines or CI/CD environments — only from within the GKE VPC.

Current state:
- 25 goose v3 migration files embedded in the Go binary via `//go:embed`
- App runs `RunMigrations()` at startup using IAM SA user — blocked by `CREATE` denial on `public` schema
- `@pulumi/postgresql` provider code exists but cannot execute due to PSC connectivity
- External Secrets Operator (ESO) syncs secrets from GCP Secret Manager to K8s Secrets
- ArgoCD manages GitOps deployment from `cloud-provisioning` repo

## Goals / Non-Goals

**Goals:**
- Replace goose app-startup migrations with Atlas Operator running inside GKE (PSC-accessible)
- Create `app` schema and grant IAM SA permissions via migration files (run as `postgres` user)
- Keep migration files in the backend repo as single source of truth
- K8s manifests for Atlas live in backend repo (migration is an app concern, not infra)
- Atlas Operator installation managed in cloud-provisioning repo

**Non-Goals:**
- Using Atlas Cloud (SaaS) — all self-hosted via ConfigMap
- Multi-schema or schema-per-tenant architecture
- Changing existing migration SQL content (clean start, no data exists)
- Atlas declarative mode (`AtlasSchema`) — using versioned `AtlasMigration` only

## Decisions

### Decision 1: Atlas Operator with versioned migrations via ConfigMap

**Choice**: Deploy Atlas Kubernetes Operator and use `AtlasMigration` CRD with migration files stored in a ConfigMap generated by Kustomize.

**Why**: The Operator runs inside the GKE cluster where PSC connectivity works. ConfigMap avoids Atlas Cloud dependency. Versioned migrations (not declarative) aligns with the team's existing SQL-first workflow.

**Alternatives considered**:
- Pulumi `@pulumi/postgresql` provider: Cannot connect to PSC-only Cloud SQL from outside the VPC
- Cloud SQL Studio manual SQL: Not IaC, not reproducible
- goose in init container: Would still run as IAM SA (no `CREATE SCHEMA` privilege)

### Decision 2: postgres user with password authentication for Atlas Operator

**Choice**: Set a password on the Cloud SQL `postgres` user via Pulumi (`gcp.sql.User` with `password` field), store it in Secret Manager, sync to K8s Secret via ESO.

**Why**: The `postgres` user has `cloudsqlsuperuser` role with `CREATEROLE`, `CREATEDB`, and `LOGIN`. It can `CREATE SCHEMA` and `GRANT` — capabilities the IAM SA lacks. Password authentication is the only option for built-in users on Cloud SQL (IAM auth is for IAM users only).

**Flow**:
1. Pulumi config → `gcp.sql.User` password → Secret Manager
2. ESO → K8s Secret `atlas-db-credentials` in `atlas-operator` namespace
3. AtlasMigration CRD → `passwordFrom.secretKeyRef` + individual credential fields (`scheme`, `host`, `port`, `user`, `database`, `parameters`) → connects as postgres

### Decision 3: Migration files in backend repo with ArgoCD multi-source

**Choice**: Place AtlasMigration CRD, ConfigMap generator, and overlays in `backend/k8s/atlas/`. ArgoCD syncs from the backend repo.

**Why**: Migrations are an application concern — developers create migration files alongside code changes. Keeping CRD + ConfigMap in the same repo means a single PR updates both the migration SQL and the K8s manifest. ArgoCD's multi-source Application pattern supports tracking multiple repos.

**Structure**:
```
backend/
  k8s/atlas/
    base/
      kustomization.yaml      # configMapGenerator (disableNameSuffixHash) + AtlasMigration
      atlas-migration.yaml    # AtlasMigration CRD (sync-wave: "1")
      migrations/             # Versioned SQL files + atlas.sum
    overlays/dev/
      kustomization.yaml      # patches host to Cloud SQL PSC endpoint
```

### Decision 4: Clean start — rewrite migrations without goose headers

**Choice**: Convert existing 25 goose migration files to plain SQL (remove `-- +goose Up/Down` headers). Since no tables exist in the target database, this is a clean slate.

**Why**: Atlas uses its own versioning system (`atlas.sum`). goose headers are unnecessary and may confuse Atlas. With no existing data, there's no backward compatibility concern.

### Decision 5: Remove goose from backend application

**Choice**: Remove `RunMigrations()` from app startup, remove goose dependency, remove `NewMigrationDB()` from rdb package.

**Why**: Atlas Operator handles migration execution. Running migrations in both places would cause conflicts. The app should only connect and serve, not manage schema.

### Decision 6: ArgoCD sync wave for migration ordering

**Choice**: Use ArgoCD `argocd.argoproj.io/sync-wave` annotations. AtlasMigration gets a lower sync wave than the backend Deployment.

**Why**: Ensures migrations complete before the app starts. ArgoCD's built-in wave mechanism is the standard approach for ordered deployments.

## Risks / Trade-offs

- **[Risk] Atlas Operator is a new dependency** → Apache 2.0 licensed, actively maintained by Ariga. Minimal resource footprint. Can be removed if needed without affecting data.
- **[Risk] postgres password management** → Password stored in Secret Manager with per-secret IAM bindings. ESO syncs with 1h refresh. Rotation requires Pulumi config update + ESO refresh.
- **[Risk] ConfigMap size limit (1MB)** → 25 migration files are well under 1MB. Monitor as migrations grow over time.
- **[Trade-off] Migration files in 2 ArgoCD sources** → Atlas Operator Helm in cloud-provisioning, AtlasMigration CRD in backend. Clear separation of concerns (infra vs app).
- **[Trade-off] goose removal is irreversible** → No data exists, clean start. If Atlas Operator is removed later, a new migration tool can be adopted without data loss.

## Implementation Notes

Lessons learned during deployment (for future reference):

- **Kustomize `configMapGenerator`**: Appends a hash suffix to ConfigMap names by default. CRD fields like `configMapRef.name` are NOT auto-rewritten by Kustomize (only native K8s refs like `volumes`, `envFrom`). Use `generatorOptions.disableNameSuffixHash: true`.
- **Cloud SQL SSL**: Cloud SQL enforces SSL even via PSC. The AtlasMigration `sslmode` parameter must be `require`, not `disable`.
- **Atlas Operator `search_path`**: The CRD `parameters` map is passed as URL query params. Comma-separated values (e.g., `search_path=app,public`) are treated as a single literal schema name. Use only `search_path=app`.
- **ArgoCD sync waves**: ConfigMap (default wave 0) must be created before AtlasMigration (wave 1). Using wave `-1` for AtlasMigration causes "ConfigMap not found" errors.
