## Context

The backend uses UUIDv7 for primary keys across all tables with surrogate IDs. However, enforcement is incomplete:

| Table | ID generated by | DB DEFAULT |
|-------|----------------|------------|
| artists, venues, events, artist_official_site, push_subscriptions | App (UUIDv7) | `uuidv7()` (unused) |
| users, tickets | DB DEFAULT | `uuidv7()` |
| homes | DB DEFAULT | `gen_random_uuid()` (UUIDv4!) |
| nullifiers | DB DEFAULT | `uuidv7()` (id never used) |

Dev DB verification confirmed all existing rows have valid UUIDv7 IDs (0 non-v7 rows), and `homes` has 0 rows.

## Goals / Non-Goals

**Goals:**

- Single ID generation strategy: all surrogate IDs generated app-side via `uuid.NewV7()`
- DB-level enforcement: CHECK constraint rejects non-UUIDv7 values
- DB-level enforcement: no DEFAULT means omitting `id` in INSERT causes an immediate error
- Remove dead code: `nullifiers.id` surrogate column and the no-MBID INSERT path in artist repository

**Non-Goals:**

- Adding UUIDv7 CHECK on FK columns (e.g., `events.venue_id`) — FK integrity is sufficient
- Changing tables with composite/natural PKs (concerts, followed_artists, latest_search_logs, merkle_tree)
- Migrating existing data — all existing rows are already UUIDv7-compliant

## Decisions

### 1. CHECK constraint for UUIDv7 version bit

**Decision:** `CHECK (substring(id::text, 15, 1) = '7')`

This validates the UUID version nibble (position 15 in the hyphenated string `xxxxxxxx-xxxx-7xxx-...`). PostgreSQL's `uuid` type already validates the overall format; this adds version-specific enforcement.

**Alternative considered:** A more comprehensive check validating the variant bits (position 20 must be 8/9/a/b). Rejected as unnecessary — we control all generation and the version check is sufficient to catch accidental `gen_random_uuid()` or UUIDv4 usage.

### 2. Drop DEFAULT rather than change it

**Decision:** Remove DEFAULT entirely instead of keeping `DEFAULT uuidv7()` as a safety net.

Rationale: The whole point is to force application-side generation. A DEFAULT masks bugs where the constructor is forgotten. An immediate `NOT NULL` violation on INSERT is the desired behavior — fail fast.

**Alternative considered:** Keep DEFAULT as a fallback. Rejected because it defeats the purpose — the developer would never know their constructor was missing.

### 3. Convert nullifiers to composite PK

**Decision:** Drop the `id` column entirely, promote `(event_id, nullifier_hash)` to PK, drop the existing unique index (now redundant with PK).

Rationale: The `id` column is never read, never returned, never referenced as FK. The unique index `idx_nullifiers_event_hash` already enforces the natural key. Converting to composite PK eliminates the dead column and aligns with the project's pattern of using natural keys where appropriate (concerts, followed_artists, etc.).

### 4. Remove no-MBID artist INSERT path

**Decision:** Remove `insertArtistsNoMBIDUnnestQuery` and the corresponding branch in `ArtistRepository.Create()`. Update `Create` use case to reject artists without MBID.

Rationale: All three use-case flows (Search, ListSimilar, ListTop) already call `FilterArtistsByMBID()` which drops MBID-less artists before persisting. The no-MBID INSERT path is dead code that produces orphaned rows (1697 in dev DB). Removing it aligns the repository with the actual business invariant: every artist must have an MBID.

### 5. Entity constructor pattern

**Decision:** Add `NewUser()`, `NewTicket()`, `NewHome()` constructors following the established `NewArtist()` pattern. Each calls `newID()` (which wraps `uuid.NewV7()`).

The `newID()` helper already exists in `entity/artist.go`. It should be moved to a shared location (e.g., `entity/id.go`) since multiple entity files will use it.

## Risks / Trade-offs

**[Risk] Forgotten constructor in future code** → Mitigated by DB CHECK constraint — INSERT with non-UUIDv7 or missing ID will fail at the database level, caught immediately in development and tests.

**[Risk] Migration applied before code deploys** → No risk. The migration only drops DEFAULT and adds CHECK. Existing code that already provides app-generated IDs (artists, venues, events) is unaffected. For tables that currently rely on DEFAULT (users, tickets), the code change must deploy in the same release as the migration. Atlas Operator applies migrations before the Deployment rolls out (sync wave ordering), so the new code with constructors will be running when the next INSERT happens.

**[Risk] Orphaned artist rows (1697 with NULL MBID)** → These rows are not referenced by concerts or followed_artists (verified). They will be cleaned up by a DELETE in the migration. If any have unexpected FK references, the migration will fail safely (FK constraint violation).

**[Trade-off] Slightly more verbose entity creation** → Every caller must use a constructor instead of struct literal. This is acceptable — it enforces the invariant at the language level and matches existing patterns.

## Migration Plan

Single migration file, applied atomically:

1. `DELETE FROM artists WHERE mbid IS NULL` — clean up orphaned rows
2. `ALTER TABLE artists ALTER COLUMN mbid SET NOT NULL` — enforce MBID requirement
3. Drop `insertArtistsNoMBIDUnnestQuery` index: `DROP INDEX IF EXISTS idx_artists_mbid` and recreate as non-partial unique index (since all rows now have MBID)
4. For each surrogate-ID table: `ALTER TABLE <t> ALTER COLUMN id DROP DEFAULT`
5. For each surrogate-ID table: `ALTER TABLE <t> ADD CONSTRAINT chk_<t>_id_uuidv7 CHECK (substring(id::text, 15, 1) = '7')`
6. For `nullifiers`: drop `id` column, drop `idx_nullifiers_event_hash`, add composite PK

Rollback: Reverse migration that restores DEFAULT and drops CHECK constraints. Artist MBID data deletion is not reversible, but those rows are confirmed unused.
