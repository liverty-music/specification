## Context

The backend follows Clean Architecture with usecase-layer orchestrators that coordinate entity repositories and external services. All usecase files have test coverage except `ticket_email_uc.go` (193 lines, HIGH priority) and `artist_image_sync_uc.go` (115 lines, MEDIUM priority). The project uses testify/assert+require, mockery-generated mocks, and table-driven tests as its standard testing conventions.

## Goals / Non-Goals

**Goals:**
- Achieve unit test coverage for all public methods in `ticket_email_uc.go` and `artist_image_sync_uc.go`
- Cover both happy-path and error-path scenarios for each method
- Use existing mockery-generated mocks; generate any missing mocks if needed
- Follow the project's established table-driven test patterns

**Non-Goals:**
- Integration tests (repository layer has its own integration tests)
- Refactoring production code (that is handled by `extract-entity-domain-logic`)
- Achieving 100% line coverage -- focus on meaningful behavioral scenarios
- Testing `mapParsedToJourneyStatus` directly (being moved to entity layer by parallel change)

## Decisions

### Decision 1: Test the usecase methods that call `mapParsedToJourneyStatus`, not the function itself

**Choice**: Write tests for `Update()` and `determineJourneyStatus()` which invoke `mapParsedToJourneyStatus`, verifying the orchestration behavior rather than the mapping logic in isolation.

**Rationale**: The `extract-entity-domain-logic` change is moving `mapParsedToJourneyStatus` to `ParsedEmailData.JourneyStatus()` on the entity. Testing the usecase orchestration ensures the integration seam works regardless of where the mapping logic lives.

### Decision 2: Use existing mock infrastructure

**Choice**: Leverage mockery-generated mocks that already exist for `TicketEmailRepository`, `TicketJourneyRepository`, `TicketEmailParser`, `ArtistRepository`, `ArtistImageResolver`, and `LogoImageFetcher`.

**Rationale**: The project has an established mock generation pipeline. Creating hand-written mocks would diverge from conventions. Any missing mocks can be generated with `mockery`.

### Decision 3: Table-driven tests with descriptive scenario names

**Choice**: Use Go table-driven tests with scenario names matching the spec requirements (e.g., "Create with valid lottery info email", "Update with non-existent ID").

**Rationale**: Consistent with every other `_test.go` file in the usecase package. Descriptive names make test failures immediately actionable.

## Risks / Trade-offs

**[Risk] Mock drift if interfaces change** -- Mitigation: Mocks are auto-generated by mockery from interface definitions. Running `mockery` before testing ensures mocks stay in sync.

**[Risk] Parallel change to `mapParsedToJourneyStatus`** -- Mitigation: Tests target the usecase methods (`Update`, `determineJourneyStatus`) rather than the function being moved. After the entity extraction, these tests will exercise the entity method through the usecase, which is the correct integration boundary.

**[Trade-off] No integration-level coverage added** -- Acceptable because repository integration tests already exist separately. Usecase tests with mocked dependencies are sufficient to verify orchestration logic.
