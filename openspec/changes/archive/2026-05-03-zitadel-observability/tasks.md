## 1. PR-1 — cloud-provisioning: enable Zitadel OTLP metrics pipeline + Cloud Monitoring AlertPolicies + dashboard panel

Goal: ship the 3 detection signals (Zitadel API latency p99, backend JWT-validation error rate, Cloud SQL connection-pool dashboard) before the band-aid CronJob. Because Zitadel v4 has no `/debug/metrics` HTTP endpoint, this PR also extends the OTLP collector with a `metrics` pipeline and configures Zitadel to push OTLP metrics to it (see design D2 — recon during the apply phase confirmed `/debug/metrics` returns 404 on v4.14.0).

### 1a. Enable OTLP metrics pipeline (prerequisite for the alerts)

- [x] 1.1 Extend `k8s/namespaces/otel-collector/base/configmap.yaml` to add a `metrics` pipeline alongside the existing `traces` pipeline. New shape: `pipelines.metrics: { receivers: [otlp], processors: [batch], exporters: [googlecloud] }`. The `googlecloud` exporter requires no extra config beyond what `traces` already uses (Workload Identity-driven auth). *(Shipped in PR-1a #220, merged 2026-05-03.)*
- [x] 1.2 Add OTEL env vars to the Zitadel API container's configmap (`zitadel-config`, generated by Pulumi `KubernetesComponent`). Required: `OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector.otel-collector.svc.cluster.local:4317`, `OTEL_EXPORTER_OTLP_PROTOCOL=grpc`, `OTEL_METRICS_EXPORTER=otlp`, `OTEL_RESOURCE_ATTRIBUTES=service.name=zitadel,service.version=v4.14.0`. Optionally `OTEL_METRIC_EXPORT_INTERVAL=60000` to make D4's 60s window explicit. *(Shipped in PR-1a #220.)*
- [x] 1.3 After 1.1 and 1.2 land + ArgoCD reconciles + a fresh Zitadel pod restarts, snapshot the OTEL-pushed metrics observable in Cloud Monitoring. Use `gcloud monitoring metrics list --filter="metric.type:opentelemetry"` (or the equivalent Cloud Console search) to capture the actual metric type name produced by the `googlecloud` exporter. Save to `tmp/zitadel-otel-metric-names.txt`. This resolves design Open Question Q2 — the alert query in 1.5 uses the confirmed name. *(Snapshot via collector logs after manual rollout: `workload.googleapis.com/rpc.server.duration` (with labels `rpc_service`, `rpc_method`, `rpc_system`) — primary signal for the latency alert. Also saw `workload.googleapis.com/db.pool.{active,idle}_connections` from Zitadel's pgx pool — bonus connection-pool signal beyond the Cloud SQL-side `num_backends` metric.)*

### 1a-fix. Post-PR-1a remediation (newly discovered during recon)

- [x] 1.1f Grant the otel-collector's Workload Identity GSA the `roles/monitoring.metricWriter` IAM role on the dev project. The existing role set covers traces (cloudtrace.spans.create) but not metrics (`monitoring.metricDescriptors.create` is required, currently denied). Manifests as `error PermissionDenied desc = Permission monitoring.metricDescriptors.create denied (or the resource may not exist)` in the collector logs after PR-1a deploy — Zitadel pushes are received and translated, but the exporter cannot create the custom metric descriptor in Cloud Monitoring. To verify the existing role binding location: `grep -rn 'otel-collector\|otel_collector' src/gcp/` and add the new role on the same `gcp.projects.IAMMember` (or whichever resource manages the collector SA's role list). *(Shipped in PR-1a-fix #222, merged 2026-05-03. Verified post-merge: `gcloud projects get-iam-policy liverty-music-dev` shows `otel-collector@…` bound to `roles/monitoring.metricWriter`. Metric `workload.googleapis.com/rpc.server.duration` registered in Cloud Monitoring.)*
- [x] 1.2f Add `reloader.stakater.com/auto: "true"` annotation to `k8s/namespaces/otel-collector/base/deployment.yaml`. The `zitadel` Deployment has this annotation and rolls automatically on configmap change, but `otel-collector` does not — when PR-1a updated the collector configmap, the pod did not restart and metrics drop persisted. Fixed manually for this rollout via `kubectl rollout restart deployment/otel-collector`, but the annotation is the durable fix so future pipeline edits (adding receivers, processors, etc.) auto-apply. *(Shipped in PR-1a-fix #222.)*
