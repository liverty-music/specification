## Deployment-incident postscript (2026-05-20 prod cutover)

Four surprises hit during the prod cutover; tracking here so the
archived change is a faithful record:

### A. ArgoCD pre-install hooks blocked by Secret rename
The chart's setup Job (`helm.sh/hook: pre-install,pre-upgrade`) ran
BEFORE ArgoCD applied the renamed ExternalSecret. The setup Job
failed with `secret "zitadel-masterkey" not found` and gated all
subsequent sync. **Mitigation:** manually `kubectl apply -f` the new
`external-secret.yaml` to unblock the hook, then ArgoCD continued.
**Future:** if renaming a chart-dependency Secret, apply the new
ExternalSecret out-of-band before merging the chart-side change.

### B. Old Deployment selector immutable + container-name collision
The pre-chart `zitadel-api` Deployment (selector `app: zitadel,
component: api`, containers including `cloud-sql-proxy`) couldn't be
patched in place to the new chart shape (selector
`app.kubernetes.io/*`, `cloud-sql-proxy` now in `initContainers` —
which violates K8s unique-name-across-container-lists rule). ArgoCD
errored: "Deployment is invalid: spec.selector immutable + duplicate
container name". **Mitigation:** `kubectl delete deploy zitadel-api`,
then ArgoCD recreated fresh with the chart shape. **Future:** when a
chart migration introduces a selector change or moves containers
between `containers`/`initContainers`, delete the old Deployment
before the merge syncs (or set ArgoCD `Replace=true` on the
Deployment).

### C. Login UI PAT mount path mismatch
The chart's auto-generated `login-config-dotenv` ConfigMap sets
`ZITADEL_SERVICE_USER_TOKEN_FILE=/login-client/pat`, and the Login
UI Next.js app reads from the `.env` file at startup, NOT process
env vars. Our initial mount at `/var/run/zitadel/login-client.pat`
(matching `login.env` inline override) was silently ignored — the
app waited for `/login-client/pat` that didn't exist. **Mitigation:**
hotfix PR #298 remounted Secret at `/login-client/pat` with subPath
+ defaultMode: 0400.

### D. `NEXT_PUBLIC_BASE_PATH` is build-time, not runtime → URL collapse reverted
We attempted to collapse `/ui/v2/login/login` → `/ui/v2/login` by
setting `NEXT_PUBLIC_BASE_PATH=/ui/v2` at runtime on the Login UI
Pod + matching probe paths + HTTPRoute prefix + `LoginV2.BaseURI`.
The override was silently ignored: Next.js inlines `NEXT_PUBLIC_*`
env vars at IMAGE BUILD time, so the v4.14.0 `zitadel-login` image
served at its baked-in `/ui/v2/login/*` regardless of our runtime
override. Our probes at `/ui/v2/{healthy,ready}` got 404 → liveness
kill → CrashLoop with 70+ restarts. **Mitigation:** revert PR #299
restored chart-default probe paths + HTTPRoute prefix + dropped
`LoginV2.BaseURI`. User-visible URL stays `/ui/v2/login/login` (the
redundant double-/login). The `SetInstanceFeatures` API calls in
§9.3a / §10.3a were therefore NOT made — those steps remain
checked because the original intent (URL collapse) was abandoned,
not because we made the API call.

### URL collapse follow-up
Tracked as a future change: either rebuild the upstream
`zitadel-login` image with a different `NEXT_PUBLIC_BASE_PATH` set
at build time, OR add a Gateway URLRewrite mapping the visible
`/ui/v2/login` to the served `/ui/v2/login/login` server-side. Both
approaches are non-trivial; pre-service-in the redundant URL is
acceptable.
