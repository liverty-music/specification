## Why

The GSM secret `zitadel-machine-key` was named during early Zitadel-Cloud-era provisioning when "machine key" referred to the only Zitadel service-user key the platform managed. After the self-hosted cutover, the platform now manages **two** Zitadel service-user keys with distinct lifecycles and consumers:

1. `zitadel-admin-sa-key` — bootstrap admin key, generated by Zitadel itself on first boot, written to GSM by the in-pod `bootstrap-uploader` sidecar, consumed by Pulumi to provision Project / Application / Actions v2 resources.
2. `zitadel-machine-key` — a Pulumi-provisioned `MachineUser` + `MachineKey` for the `backend-app` service user, consumed by the **backend** application to call Zitadel APIs (e.g., create users at sign-up time, resend verification emails).

The current name `zitadel-machine-key` is ambiguous — neither callers reading code nor operators reading GSM see at a glance which of the two service users it represents. During the §13.15 incident chain (`fix(zitadel): refresh backend-app MachineKey to align with self-hosted`), this ambiguity directly cost time: the operator had to grep call sites to confirm which Zitadel principal the GSM secret was bound to before force-replacing it.

Renaming to `zitadel-backend-app-key` makes the binding explicit and matches the Pulumi resource name (`backend-app` MachineUser).

## What Changes

- **The GSM secret `zitadel-machine-key` SHALL be renamed to `zitadel-backend-app-key`** via a zero-downtime split:
  1. Pulumi adds the new secret alongside the old, populated from the same MachineKey output.
  2. Backend's `ExternalSecret` is updated to read both names (transitional: prefer new, fall back to old).
  3. Pulumi removes the old secret writer.
  4. Backend's `ExternalSecret` drops the old-name fallback; reads only the new name.
  5. The orphaned `zitadel-machine-key` GSM secret is destroyed.
- **Cross-repo coordination order** SHALL be enforced via change tasks: cloud-provisioning step 1 (add new secret) MUST be deployed before backend step 2 (read both); backend step 4 (read new only) MUST be deployed before cloud-provisioning step 5 (destroy old). Each step is a separate PR with the dependency declared in its description.
- **Pulumi resource names** in `src/zitadel/components/machine-user.ts` MAY be renamed in the same change for symmetry (`backend-app-machine-key` → matches `zitadel-backend-app-key`), but this is opt-in: the resource rename forces a single replace cycle, so it MAY be deferred to the same window as the §18.7 K8s naming cleanup.

## Capabilities

### Modified Capabilities

- `identity-management`: The "Backend MachineKey Lifecycle Tied to Zitadel-Side Identity" requirement SHALL be updated to use the new GSM secret name. The lifecycle requirement itself does not change; only the name changes.

### New Capabilities

None.

### Removed Capabilities

None.

## Impact

**Affected repositories**

- `cloud-provisioning/`: `src/zitadel/components/machine-user.ts` (writes both names during transition), `src/zitadel/components/secrets.ts` (provisions both GSM resources during transition), `k8s/namespaces/backend/base/external-secret.yaml` (reads both during transition).
- `backend/`: no code change. The backend reads its Zitadel SA key from the K8s Secret synced by ESO, not directly from GSM. The K8s Secret's keys MAY change name; if they do, `internal/infrastructure/zitadel/auth.go` (or wherever the env var binding lives) needs the env var name updated.

**Resources**

- New GSM secret `zitadel-backend-app-key` is provisioned alongside the existing `zitadel-machine-key`. After full cutover, only the new one remains.
- ESO `ExternalSecret` is updated twice (read-both → read-new-only) — cleaner with a single transitional step but worth the operational simplicity.

**Reversibility**

- Each PR is reversible by reverting the prior PR. The old GSM secret stays alive until the final cleanup PR, so any rollback before that PR is a single revert.
- Once the final cleanup PR is merged, rollback requires re-creating the old GSM secret — possible but tedious. Mitigation: the cleanup PR SHALL be merged only after the new-only PR has soaked for at least 7 days.

**Out of scope**

- Renaming `zitadel-admin-sa-key` (already correctly named).
- Renaming the K8s Service Account `zitadel/zitadel` (orthogonal — covered by `k8s-naming-cleanup`).
- Renaming the Pulumi resource `backend-app-machine-key` if that forces a Zitadel-side MachineKey replace (out of scope unless co-bundled).

**Dependencies**

- Requires `self-hosted-zitadel` to be archived (or actively archiving) so this change applies to a stable spec base.
- Independent of `archive-zitadel-cloud-tenant`, `k8s-naming-cleanup`, `pulumi-deploy-safeguards`.
