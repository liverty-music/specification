# atlas-operator Specification

## Purpose

Defines the requirements for database schema migration management using the Atlas Kubernetes Operator. The operator runs inside GKE where PSC connectivity to Cloud SQL is available, replacing the previous application-startup migration approach.

## Requirements

### Requirement: Atlas Operator SHALL be installed in the GKE cluster

The system SHALL deploy the Atlas Kubernetes Operator via Helm chart in a dedicated namespace. The Operator SHALL manage `AtlasMigration` CRDs to apply versioned database migrations.

#### Scenario: Operator deployment

- **WHEN** ArgoCD syncs the atlas-operator namespace
- **THEN** the Atlas Operator controller pod SHALL be running
- **AND** the `AtlasMigration` CRD SHALL be registered in the cluster

### Requirement: Migrations SHALL be stored as ConfigMap

The system SHALL store migration SQL files as a Kubernetes ConfigMap generated by Kustomize `configMapGenerator` from the backend repository's migration directory. The ConfigMap SHALL include an `atlas.sum` integrity file.

#### Scenario: ConfigMap generation from migration files

- **WHEN** `kubectl kustomize` renders the backend atlas overlay
- **THEN** a ConfigMap SHALL be generated containing all migration SQL files
- **AND** the ConfigMap SHALL include the `atlas.sum` hash file
- **AND** file names SHALL be preserved as ConfigMap keys

### Requirement: AtlasMigration CRD SHALL connect as postgres user

The `AtlasMigration` CRD SHALL connect to Cloud SQL using the `postgres` built-in user with password authentication. The password SHALL be sourced from a Kubernetes Secret synced by External Secrets Operator from GCP Secret Manager.

#### Scenario: Database connection via Secret reference

- **WHEN** the Atlas Operator reconciles an AtlasMigration resource
- **THEN** it SHALL read the database URL from a Kubernetes Secret via `urlFrom.secretKeyRef`
- **AND** the URL SHALL use `postgres` user with password authentication
- **AND** the connection SHALL route through PSC (Private Service Connect)

### Requirement: Migrations SHALL create the app schema and grant permissions

The first migration file SHALL create the `app` schema, grant `CREATE` and `USAGE` privileges to the backend IAM service account, and set default privileges for tables and sequences.

#### Scenario: Schema bootstrap migration

- **WHEN** Atlas applies the first migration
- **THEN** the `app` schema SHALL be created
- **AND** `CREATE` and `USAGE` SHALL be granted to the IAM SA on the `app` schema
- **AND** default privileges SHALL grant `ALL` on tables and sequences to the IAM SA

### Requirement: Backend repository SHALL host an Atlas migration prod overlay

The `liverty-music/backend` repository SHALL contain a `k8s/atlas/overlays/prod/` directory whose `kustomization.yaml` renders the prod-flavored Atlas `AtlasMigration` resources for the `liverty-music-prod` Cloud SQL instance. The directory's existence is a precondition for the corresponding ArgoCD Application (`backend-migrations` in `cloud-provisioning/k8s/argocd-apps/prod/`) to sync: without the path, ArgoCD reports `Status=Unknown (ComparisonError)` because Kustomize manifest generation fails.

The prod overlay SHALL diverge from the existing dev overlay in:

1. Target Cloud SQL instance — the `AtlasMigration.spec.cloud.url` (or equivalent connection string) SHALL reference the prod PSC DNS name (`298474959c18.25pf3r4b6sfkn.asia-northeast2.sql.goog`) and database `liverty-music`.
2. IAM database user — the Atlas connection SHALL authenticate as `backend-app@liverty-music-prod.iam` (Atlas Operator currently uses the backend-app GSA for migration apply, mirroring dev's pattern).
3. Migration source ConfigMap — generated by `configMapGenerator` from the same `migrations/` directory as dev (no fork; both envs apply the same forward migrations).

#### Scenario: Prod overlay directory exists

- **WHEN** listing `liverty-music/backend:k8s/atlas/overlays/`
- **THEN** the directory SHALL contain both `dev/` and `prod/` subdirectories
- **AND** each SHALL contain a `kustomization.yaml`

#### Scenario: Prod overlay renders cleanly

- **WHEN** running `kubectl kustomize k8s/atlas/overlays/prod` from the `liverty-music/backend` repo root
- **THEN** the command SHALL exit with code 0
- **AND** SHALL emit at least one `AtlasMigration` resource targeting the prod Cloud SQL connection

#### Scenario: ArgoCD prod backend-migrations Application reaches Synced/Healthy

- **WHEN** the `backend-migrations` ArgoCD Application in the prod cluster polls its source (`liverty-music/backend:k8s/atlas/overlays/prod` on `main`)
- **THEN** the Application's `status.sync.status` SHALL transition from `Unknown` to `Synced`
- **AND** `status.health.status` SHALL remain `Healthy`
- **AND** no `ComparisonError` condition SHALL be present
